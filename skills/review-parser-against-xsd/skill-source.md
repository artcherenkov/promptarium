# Review Parser Against XSD

## Когда Использовать

Используй этот скилл, когда нужно честно ответить, соответствует ли парсер XSD-схеме.

Он подходит для строгого ревью парсера, аудита потери контекста (`ctx`), проверки диагностик (`diagnostics`), глоссария (`glossary`), форматирования ошибок через `formatDiagnostic` и любых других признаков корректности парсера относительно XSD.

Если в кодовой базе есть дополнительные слои, которые влияют на результат парсинга, диагностики, вывод CLI или публичный контракт, включай их в ревью даже тогда, когда они не названы в этом скилле.

## Главный Принцип

Доказывай соответствие парсера XSD через наблюдаемое поведение.

Используй чтение кода, тесты и типизацию как вспомогательные сигналы. Основной вес давай цепочке: XSD задаёт правило, `xmllint` проверяет XML, публичная точка запуска парсера показывает фактическое поведение, диагностики объясняют расхождение.

## Рабочий Подход

1. Определи область проверки: документ, версия, XSD, XML-примеры, публичная точка запуска, затронутые файлы и пользовательский вопрос.
2. Найди актуальную архитектуру парсера в текущем коде, а не только известные старые пути.
3. Проверь валидные XML-примеры через `xmllint`.
4. Запусти парсер на тех же XML через публичную точку запуска.
5. Создай маленькие испорченные XML-примеры под конкретные риски.
6. Сравни поведение XSD и парсера.
7. Проверь, что диагностики имеют точный путь, код ошибки и полезные детали.
8. Проследи путь результата дальше: форматирование, глоссарий, CLI-вывод, публичные типы и другие актуальные слои.
9. Сформулируй вердикт и явно назови границы проверки.

Этот порядок можно расширять. Если кодовая база подсказывает дополнительный важный слой, команду, формат вывода или риск, проверь его сверх базового маршрута.

## Журнал Проверок

Веди журнал проверок в отдельном временном файле, чтобы по нему позже можно было написать интеграционные тесты.

Создай файл в `/private/tmp`, например:

/private/tmp/xsd-strict-review-<doc>-<version>-<timestamp>.md

Записывай туда каждую существенную проверку:

- цель проверки;
- путь к XSD;
- путь к исходному XML;
- путь к испорченному XML, если он создан;
- команда `xmllint`;
- результат `xmllint`;
- команда запуска парсера;
- результат парсера;
- ожидаемая диагностика;
- фактическая диагностика;
- вывод: совпадает с XSD, расходится с XSD, проверка неполная.

Пиши журнал так, чтобы из него можно было восстановить тест: входной XML, ожидаемый результат и смысл проверки.

В финальном ответе дай ссылку на файл журнала. Если журнал не удалось создать, явно скажи почему.

## Формат Ответа

Начинай с одной строки:

- `approved` — существенных расхождений с XSD не найдено;
- `not approved` — есть доказанные расхождения;
- `нельзя честно утверждать соответствие XSD` — проверка неполная или остался существенный риск.

После вердикта перечисли замечания по важности. Для каждого замечания дай файл и строку, факт из XSD, результат проверки, поведение парсера и риск.

Если часть проверок не удалось выполнить, назови этот пробел и объясни, как он влияет на уверенность.

В конце ответа дай ссылку на временный файл с журналом проверок.

## Что Считать Сильным Доказательством

Хорошее доказательство строится как цепочка:

1. XSD задаёт правило.
2. `xmllint` подтверждает или отвергает XML по этому правилу.
3. Парсер ведёт себя так же или расходится с XSD.
4. Диагностика содержит точный путь, код ошибки и полезные детали.
5. Пользовательский вывод сохраняет смысл ошибки.

Если цепочка обрывается, укажи, где именно не хватает данных.

## Что Смотреть В XSD

Смотри всё, что влияет на допустимость XML и форму результата. Начни с этих пунктов, но расширяй список по реальной схеме:

- корневой элемент;
- значение и тип `Version`;
- `sequence` и `choice`;
- `minOccurs` и `maxOccurs`;
- обязательные и повторяемые поля;
- enum-значения;
- `pattern`;
- `fractionDigits`;
- числовые типы `xs:decimal`, `xs:integer`, `xs:positiveInteger`;
- вложенные named types;
- любые ограничения, которые парсер может случайно принять или потерять.

## Что Смотреть В Коде

Иди от публичного поведения к внутренним деталям.

Начни с текущей публичной точки запуска, маршрутизации и семейного парсера. Затем проверь helper-функции, диагностики, форматирование результата, глоссарий, CLI-вывод и тесты.

Типовые места для validator:

- `packages/parser/src/parse.ts`;
- `packages/parser/src/router.ts`;
- `packages/parser/src/families/...`;
- `packages/parser/src/kernel`;
- `packages/parser/src/format`;
- `packages/parser/src/glossary`;
- `packages/cli/src`;
- `packages/parser/tests/families/...`;
- `packages/cli/tests`.

Если текущая архитектура изменилась, следуй фактическому коду, а не этому списку.

## Контекст ctx

Проверяй, что ошибки записываются в правильном месте дерева.

Особенно полезно смотреть вызовы `required*`, `optional*`, чтение простых полей, массивы, контейнеры, `sequence`, `choice` и повторы. Если есть сомнение, создай XML с вложенным элементом, дублем или неверным типом и сравни результат с `xmllint`.

## Числа

Для `xs:decimal`, `xs:integer` и `xs:positiveInteger` проверяй не только успешный парсинг, но и сохранение смысла значения.

Полезные случаи: большие значения, ведущий `+`, ведущие нули, дробная часть, `fractionDigits`, enum-значения и значения около границ. Если XSD допускает значение, парсер должен сохранить его без потери точности. Если XSD запрещает значение, парсер должен дать диагностику.

## Испорченные XML

Создавай маленькие XML-примеры под один риск за раз.

Примеры полезных мутаций:

- удалить обязательное поле;
- продублировать одиночное поле;
- переставить элементы в `sequence`;
- смешать ветки `choice`;
- превысить `maxOccurs`;
- вставить вложенный XML в простое поле;
- передать enum-значение вне списка;
- передать неверный GUID;
- передать число с пограничной лексической формой.

Храни временные примеры в `/private/tmp`, если пользователь не просит добавить их в репозиторий.

## Полезные Команды

Проверка XML против XSD:

xmllint --noout --schema references/schemas/v201/<DOC>.xsd references/data/v201/valid/<file>.kenml

Запуск парсера через CLI:

/Users/a.cherenkov/.bun/bin/bun run packages/cli/src/index.ts parse <file>

Проверка parser package:

/Users/a.cherenkov/.bun/bin/bun run --cwd packages/parser check

/Users/a.cherenkov/.bun/bin/bun run --cwd packages/parser test

Проверка CLI package:

/Users/a.cherenkov/.bun/bin/bun run --cwd packages/cli check

/Users/a.cherenkov/.bun/bin/bun run --cwd packages/cli test

Если `bun` доступен через `PATH`, можно использовать `bun`. Если команды в проекте изменились, используй актуальные scripts из `package.json`.

## Планка Для approved

Ставь `approved`, когда выполнены условия:

- валидные XML проходят `xmllint`;
- те же XML принимает публичная точка запуска парсера;
- ключевые ограничения XSD проверены на положительных и отрицательных примерах;
- испорченные XML дают диагностики;
- диагностики имеют полезные path/code/detail;
- пользовательский вывод сохраняет смысл ошибки;
- оставшиеся непроверенные зоны несущественны для заявленной области проверки.

Если условие не выполнено, выбери `not approved` или `нельзя честно утверждать соответствие XSD` и назови, какие проверки нужны для уверенности.
